| Area | Current Code | Required Change | Risk | Recommended Action |
|---|---|---|---|---|
| 1. Existing keyword-related entities | `Keyword` is subject-scoped (`subjectId`, unique `(subjectId, name)`) in `src/keywords/entities/keyword.entity.ts`. `DocumentKeyword` mapping entity exists in `src/keywords/entities/document-keyword.entity.ts`. `DocumentEntity` has `documentKeywords` relation in `src/document/entities/document.entity.ts`. | Redesign keyword ownership to document-scoped: `keywords.document_id` FK to `documents.id`, unique `(document_id, name)`. Remove `DocumentKeyword` entity and all `documentKeywords` relations. | Existing data/relations are built around subject-level reuse; direct switch can break reads/writes and existing migration history. | Introduce phased migration (backfill + dual-read window if needed), then fully remove mapping-based model. |
| 2. Existing `document_keywords` usage | `document_keywords` is actively used for read/write in `KeywordsService` (`upsert`, `find`) and `DocumentService` (`find`, `delete`), and registered in `KeywordsModule`/`DocumentModule`. Also created by migration `1744100000000-CreateKeywordsTables.ts` and referenced by `1744200000000-AlignDocumentIdToUuid.ts`. | Stop using `document_keywords` completely. Persist keyword metadata directly in `keywords` rows keyed by `document_id`. Replace all reads/writes/deletes currently targeting mapping rows with direct keyword table queries. | High: this table is currently the backbone for `GET /documents/:documentId/keywords`, analyze response keywords, and document delete cleanup. | First refactor services to direct `keywords` queries, then remove table/entity/module wiring, then add drop-table migration. |
| 3. Existing `mastery_scores` usage | No `mastery_scores` entity/module/service/controller exists in current `src`. No runtime mastery update pipeline exists yet. | When implementing mastery, design must use `(user_id, keyword_id)` only. Do not add/restore `subject_id` in `mastery_scores`. Subject-level mastery must be derived by joining through `keywords -> documents -> subjects`. | Medium: requirements docs still include legacy wording like "subject-level mastery"; implementers may accidentally add `subject_id` and create schema drift. | Lock schema first with explicit migration/entity for `mastery_scores` unique `(user_id, keyword_id)` and document derivation query patterns in service contracts. |
| 4. Existing `quiz_problem_keywords` usage | No `quiz_problem_keywords` entity or service usage is present in current codebase (only design docs and backlog references). | Future implementation must map `quiz_problem_id -> keyword_id` where keyword is document-scoped. Ensure lecture quiz keyword selection validates keyword ownership under the target document. | Medium: if future quiz modules are scaffolded from old subject-scoped assumptions, mapping may incorrectly accept cross-document same-name keywords. | Add validation rule early: selected `keywordIds` must belong to `documentId` in quiz creation flow. |
| 5. Existing services/controllers/DTOs affected | `KeywordsService`, `KeywordsController`, `KeywordResponseDto`, `DocumentService`, `DocumentModule`, `KeywordsModule`, and analyze/detail response shapes depend on subject-scoped keyword + `document_keywords`. `KeywordResponseDto` currently exposes `subjectId` but not `documentId`. | Update service/query contracts and DTOs to include document context (`documentId`) and remove mapping semantics. `GET /subjects/:subjectId/keywords` should aggregate via documents->keywords, not via `keywords.subjectId`. | High: API consumers may rely on current `subjectId` field and implicit dedup behavior. | Version response DTOs or apply coordinated backend/frontend rollout; update controller/service tests with document-context assertions. |
| 6. Required DB migration changes | Current migrations create `keywords(subjectId, name)` and `document_keywords(documentId, keywordId, importanceScore)` with camelCase columns and `document_entity` FK naming. No mastery/quiz-keyword tables yet. | Add migration sequence to: (1) add `keywords.document_id` and new keyword columns per spec, (2) backfill from `document_keywords`, (3) enforce unique `(document_id, name)`, (4) drop subject-based constraints/columns, (5) drop `document_keywords`, (6) create future `mastery_scores` with `keyword_id` FK only. | Very high: irreversible data loss if dropping mapping table before backfill; naming inconsistency (`document_entity` vs `documents`) increases migration fragility. | Implement idempotent backfill with integrity checks (duplicate same-name-per-document collisions, null ownership guard), then perform destructive drops only after verification queries pass. |
| 7. Required TypeORM entity changes | `Keyword` entity shape is incompatible with spec (subject-scoped, minimal fields). `DocumentKeyword` exists but should not. `DocumentEntity` table name is default `document_entity`, not explicit `documents`, and includes prototype parser-job fields. | Redefine entities to spec-aligned ownership and names: `DocumentEntity` mapped to `documents`, `Keyword` mapped with `document_id` FK and spec fields, remove `DocumentKeyword` entity completely. Future `MasteryScore` entity must reference `keyword_id` only. | High: renaming table mappings without synchronized migration causes runtime query failures. | Apply entity changes together with migrations in one release unit; disable partial deploys where entity and DB shape mismatch. |
| 8. Required API changes | Current keyword APIs are live: `GET /subjects/:subjectId/keywords`, `GET /documents/:documentId/keywords`. Current payload (`KeywordResponseDto`) lacks document identity and is derived from subject or mapping model. | Preserve endpoints but change implementation semantics and response shape to include document-scoped context (at minimum `documentId`; optionally document title). Ensure subject-level endpoint is derived aggregation, not direct subject FK filter. | Medium: frontend sorting/grouping behavior may change when same-name keywords become multiple distinct rows by document. | Update `document/08-api-interface.md` contract and implement compatibility notes for frontend grouping/display logic. |
| 9. Risky areas | Current analyze flow (`DocumentService.analyzeDocument`) writes through mapping table; document detail/status/delete also assume mapping. Subject keyword endpoint currently fetches from `keywords.subjectId`, which conflicts with new design. | Replace all keyword persistence/retrieval touchpoints in analyze/detail/list/delete flows and ensure ownership checks still work after relation changes. | Very high: regression risk in upload->analyze->keyword listing critical path. | Add integration tests for: upload, analyze, document keywords, subject keywords, delete document, and cross-document same-name keyword isolation. |
| 10. Recommended implementation order | Code is currently partially implemented for documents/keywords but not quiz/mastery domains; this is a good window to realign core schema before more modules depend on old model. | Execute in strict order: schema backfill -> entity refactor -> service refactor -> API DTO update -> cleanup/removal -> test hardening. | Medium if ordered; high if done ad hoc. | 1) Add safe migration/backfill. 2) Update entities/modules. 3) Refactor `KeywordsService` and `DocumentService` reads/writes. 4) Update controllers/DTOs and API docs. 5) Remove `DocumentKeyword` artifacts. 6) Add integration tests and run full migration verification on staging DB snapshot. |
